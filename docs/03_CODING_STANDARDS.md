# Jarvis AIOS — Coding Standards

**Version:** 1.0  
**Status:** Ratified  

---

## 1. Purpose

This document defines the non-negotiable standards for writing code in the Jarvis AIOS project.

It is not a Python tutorial. It is an engineering standards document for a production-grade AI Operating System. Every contributor — human or AI — must follow these rules.

When the Coding Standards conflict with the Engineering Constitution (`docs/00_ENGINEERING_CONSTITUTION.md`) or the Architecture (`docs/02_ARCHITECTURE.md`), those higher-level documents win.

---

## 2. Scope

These standards apply to all Python source code in the `backend/` directory.

They apply to:

- Application code (all files under `backend/app/`).
- Tests (all files under `backend/app/tests/`).
- Scripts and utilities.
- Configuration files (`.py` only).

They do **not** apply to:

- Markdown documentation.
- Frontend code (future: JavaScript/TypeScript standards will be documented separately).
- Files generated by tools (lock files, compiled assets).

---

## 3. Engineering Philosophy

Code in Jarvis is written to be **readable, correct, and maintainable** — in that order.

This philosophy flows from the Engineering Constitution:

- **Discipline over Genius** — Write boring, obvious, correct code. Clever code is a liability.
- **Explicit over Implicit** — Every dependency, every data flow, every side effect must be visible.
- **Production Quality Is Not Optional** — Logging, error handling, validation, and security are part of every feature.
- **Architecture over Convenience** — If a shortcut violates the architecture, it is not a shortcut — it is technical debt.

The architecture (`docs/02_ARCHITECTURE.md`) defines strict layer boundaries. Code that violates these boundaries is rejected regardless of how well it is written.

## AI-Assisted Development

Jarvis embraces AI-assisted software engineering.

AI tools accelerate implementation but never replace engineering judgment.

Rules:

- AI-generated code always requires human review.
- AI must follow AGENTS.md before making changes.
- AI must preserve architecture.
- AI should explain significant implementations before applying them.
- Human approval remains mandatory for architectural and security-sensitive changes.

---

## 4. Definition of Clean Code

In Jarvis, code is considered "clean" when:

1. **A function does one thing and does it well.** If a function has more than one responsibility, split it.
2. **A class abstracts one concept.** If a class has more than one reason to change, split it.
3. **A module serves one layer.** FastAPI modules serve the HTTP layer. Tool modules serve the tool layer. They do not cross.
4. **Intent is visible without running the code.** The name, types, and structure of the code tell you what it does.
5. **Code fails loudly and informatively.** Errors are raised with meaningful messages. Exceptions are never silently swallowed.
6. **It can be understood by a new engineer in under 30 seconds.** If it takes longer, it needs simplification.
7. **It is testable in isolation.** Dependencies are injected. Side effects are explicit. State is contained.

## 5. Python Standards

### 5.1 PEP 8 Compliance

All code must conform to PEP 8, except where this document specifies stricter rules.

### 5.2 Stricter Jarvis Rules

| Rule | PEP 8 Default | Jarvis Standard |
|---|---|---|
| Line length | 79 characters | 88 characters (Black-compatible) |
| Maximum function length |	Not specified	|Functions should remain concise and focused on a single responsibility. Readability is preferred over an arbitrary line limit.|
| Maximum parameters |	Not specified	| Prefer parameter objects (dataclass or Pydantic model) when a function requires many related parameters.|
| Indentation | 4 spaces | 4 spaces (no tabs) |
| Blank lines | 2 around functions, 2 around classes | Same — no deviations |

### 5.3 Python 3.12+ Features

Jarvis targets Python 3.12+. The following modern features are encouraged:

- **`match` statements** — For pattern matching on discriminated unions and state values. Prefer over long `if/elif` chains.
- **`Self` type** — Use `Self` for return types of class methods that return `self`.
- **Type parameter syntax** — Use `list[str]` over `List[str]` (3.9+), `dict[str, int]` over `Dict[str, int]`.
- **Improved error messages** — Python 3.12+ tracebacks are more informative. Use them — avoid catching and re-raising with less information.

### 5.4 Formatting

Formatting must remain consistent across the repository.

A single automated formatter (currently Black) should be used to enforce formatting.

Formatting configuration must be version controlled and applied consistently across all contributors.

## 6. Project Structure Standards

### 6.1 One File = One Concern

Each file must have a single logical concern. If a file contains two unrelated classes or functions, split it.

### 6.2 Directory Structure Mirrors Architecture

The directory tree under `backend/app/` mirrors the architectural layers defined in `docs/02_ARCHITECTURE.md`:

```
app/
├── Config/          # Environment configuration
├── Constants/       # Shared constants
├── Data/            # Data storage (future)
├── Exceptions/      # Exception hierarchy
├── FastAPI/         # HTTP layer
├── Jarvis/          # Runtime layer
├── LangGraph/       # Orchestration layer
│   └── nodes/       # LangGraph nodes
├── LLM/             # LLM client layer
├── Logs/            # Logging (future)
├── Memory/          # Memory layer
├── Models/          # Data models (future)
├── Prompts/         # Prompt templates
├── Services/        # Service layer
├── tests/           # Tests (mirrors source)
├── Tools/           # Tool system
│   ├── tool.py      # Base class
│   ├── registry.py  # Tool registry
│   ├── engine.py    # Tool engine
│   └── ...          # Individual tools
└── Utils/           # Shared utilities (future)
```

### 6.3 `__init__.py` Exports

The `__init__.py` file of each package should export only the public interface:

- Classes and functions that are part of the module's public API.
- Constants and types that consumers of the module need.

Internal implementation details must **not** be re-exported from `__init__.py`.

### 6.4 Nesting Depth

Keep directory nesting to 3-4 levels maximum:

- `app/LangGraph/nodes/` — acceptable (3 levels).
- `app/Tools/` — acceptable (2 levels).
- `app/Layer/Sub/Sub/Sub/Sub/` — unacceptable.

### 6.5 Test Structure

Test files mirror the source structure exactly:

```
app/tests/
├── test_FastAPI/
│   └── test_routes.py        # Tests for app/FastAPI/routes.py
├── test_Tools/
│   ├── test_engine.py        # Tests for app/Tools/engine.py
│   ├── test_registry.py      # Tests for app/Tools/registry.py
│   ├── test_calculator.py    # Tests for app/Tools/calculator.py
│   └── ...
└── test_LangGraph/
    └── test_graph.py          # Tests for app/LangGraph/graph.py
```

## 7. Naming Conventions

### 7.1 General Rules

| Construct | Convention | Example |
|---|---|---|
| Functions, methods | `snake_case` | `execute_tool()`, `get_memory()` |
| Variables | `snake_case` | `session_id`, `tool_name` |
| Modules, packages | `snake_case` | `tool.py`, `chat_service.py` |
| Classes, types | `PascalCase` | `CalculatorTool`, `ChatRequest` |
| Pydantic models | `PascalCase` | `ChatRequest`, `ChatResponse` |
| Constants | `UPPER_CASE` | `MAX_RETRIES`, `DEFAULT_TIMEOUT` |
| Internal / private | `_prefix` | `_validate_args()`, `_tools` |
| Name-mangled (rare) | `__prefix` | `__internal_state` |

### 7.2 Descriptive Names

Names must communicate intent. Abbreviations are allowed only for:

- Loop counters: `i`, `j`, `k`
- Exceptions: `e`
- Unused variables: `_`
- Well-known acronyms: `API`, `LLM`, `HTTP`, `REST`

Everything else must be spelled out:

```python
# Good
session_id: str
tool_registry: ToolRegistry
memory_backend: MemoryBackend

# Bad
sid: str
tr: ToolRegistry
mb: MemoryBackend
```

### 7.3 Module Names

Module names are short, singular nouns:

- `tool.py`, not `tools.py`
- `registry.py`, not `registries.py`
- `route.py`, not `routes.py`

Exception: when the module contains purely utility functions with no central class.

## 8. Imports

### 8.1 Import Style

Absolute imports are preferred.

Relative imports may be used only within the same package when they improve readability.

Wildcard imports are forbidden.

Circular imports must be solved through architectural refactoring rather than import tricks.

### 8.2 No Wildcard Imports

```python
# Forbidden
from app.Tools import *
from app.Memory import *
```

### 8.3 Import Order

Imports must be grouped and ordered as follows, with a blank line between each group:

1. **Standard library** (`os`, `sys`, `datetime`, `json`, `abc`, `pathlib`, etc.)
2. **Third-party libraries** (`fastapi`, `pydantic`, `langgraph`, `langchain`, etc.)
3. **First-party / application imports** (`app.Config`, `app.Tools`, etc.)

Within each group, imports are sorted alphabetically.

### 8.4 First-Party Import Order

Within the first-party group, order by architectural layer (top-down):

```
1. app.Config
2. app.Constants
3. app.Exceptions
4. app.FastAPI
5. app.Jarvis
6. app.LangGraph
7. app.Tools
8. app.Memory
9. app.LLM
10. app.Prompts
11. app.Services
12. app.Utils
```

### 8.5 What to Import

Import classes and functions, not modules — unless the module name provides necessary namespacing:

```python
# Good
from app.LangGraph.state import State
from app.Tools.engine import engine

# Acceptable (when namespacing adds clarity)
from app import Config
# Config.settings.API_KEY is clear

# Bad
import app.LangGraph.state
# Must write app.LangGraph.state.State everywhere
```

## 9. Type Hints

### 9.1 When Required

Type hints are **required** for:

- All function parameters and return types.
- All class attributes.
- All Pydantic model fields.
- All TypedDict fields.
- All module-level variables.

### 9.2 The `Any` Exception

`Any` is allowed only when a truly generic value cannot be typed. Every use of `Any` must be justified with a comment:

```python
# Good
def execute(self, **kwargs: Any) -> Any:
    """
    Tool arguments and return values are generic.
    Each tool defines its own schema in its docstring.
    """
    ...

# Bad
def process(data: Any) -> Any:  # No justification
    ...
```

### 9.3 Modern Syntax

Use modern type syntax (Python 3.9+):

```python
# Good
def get_tools(self) -> list[Tool]: ...
def lookup(self, name: str) -> dict[str, int]: ...
from collections.abc import Sequence  # Prefer over typing.Sequence

# Bad
from typing import List, Dict, Tuple
def get_tools(self) -> List[Tool]: ...
def lookup(self, name: str) -> Dict[str, int]: ...
```

### 9.4 Structured Data Types

Never use raw dictionaries for public interfaces. Choose the appropriate structured type:

| Use Case | Type |
|---|---|
| Request/response schemas | `pydantic.BaseModel` |
| Internal data containers | `dataclass` |
| LangGraph state | `TypedDict` |
| Configuration | `pydantic.BaseSettings` |
| Named tuples | `NamedTuple` (for simple, immutable structures) |

```python
# Good
from pydantic import BaseModel

class ChatRequest(BaseModel):
    session_id: str
    message: str

# Bad
def chat(data: dict) -> dict:
    # What keys does this dict have? Unknown.
    ...

# Bad — reinventing what TypedDict does
class State:
    def __init__(self, session_id: str, message: str):
        self.session_id = session_id
        self.message = message
```

### 9.5 Type Aliases

Use type aliases for complex, repeated type signatures:

```python
from typing import TypeAlias

ToolMap: TypeAlias = dict[str, Tool]
ActionResult: TypeAlias = dict[str, str | int | float | None]
```

## 10. Functions

### 10.1 Single Responsibility

A function must do one thing. If a function performs two distinct operations, split it:

```python
# Good
def validate_input(data: ChatRequest) -> None: ...
def process_message(message: str) -> Action: ...

# Bad
def validate_and_process(data: ChatRequest) -> Action:
    ...
```

### 10.2 Function Size

Functions should remain concise and focused on a single responsibility.

If a function becomes difficult to understand, split it into smaller responsibilities.

Readability is more important than an arbitrary line limit.

Exception:

Some data transformation functions, builders, or state construction functions may naturally be longer while still remaining readable.

### 10.3 Function Parameters

Functions should avoid excessive parameters.

When several parameters belong together, prefer a parameter object implemented with a dataclass or Pydantic model.

Parameter count should never reduce readability.

Group related values instead of passing many independent arguments.

### 10.4 Pure Functions

Prefer pure functions (no side effects) over state-mutating functions. When side effects are necessary (logging, writing to storage), make them explicit in the function name:

```python
# Good
def calculate_total(values: list[float]) -> float:  # Pure
def save_to_session(session_id: str, data: dict) -> None:  # Side effect in name

# Bad
def process(values: list[float]) -> float:
    # Silently logs something, modifies global state, or writes to disk
    ...
```

### 10.5 Boolean Parameters

Boolean parameters are a code smell. They often indicate that a function does two things:

```python
# Bad
def send_message(message: str, urgent: bool) -> None: ...

# Good
def send_normal_message(message: str) -> None: ...
def send_urgent_message(message: str) -> None: ...
```

## 11. Classes

### 11.1 When to Use a Class

Use a class when you need to encapsulate state with the behaviour that operates on that state. If a class has only one method (besides `__init__`), consider whether a function is sufficient.

### 11.2 Inheritance

Prefer inheriting from framework-provided bases rather than creating deep custom hierarchies:

- Tools inherit from `Tool(ABC)`.
- Pydantic models inherit from `BaseModel`.
- LangGraph state inherits from `TypedDict`.

Avoid custom inheritance chains deeper than 2-3 levels:

```python
# Good
class CalculatorTool(Tool): ...
class DateTimeTool(Tool): ...

# Bad
class BaseTool(Tool): ...
class MathTool(BaseTool): ...
class AdvancedMathTool(MathTool): ...
class CalculatorTool(AdvancedMathTool): ...
```

### 11.3 Composition over Inheritance

Inject dependencies through the constructor rather than subclassing:

```python
# Good
class ToolEngine:
    def __init__(self, registry: ToolRegistry):
        self._registry = registry

# Bad
class ToolEngine(RegistryConsumer):  # Implicit dependency through inheritance
    ...
```
### 11.4 Dependency Injection

Dependencies should be injected rather than created internally.

Depend on abstractions whenever practical.

This improves:

- Testability
- Maintainability
- Replaceability
- Future extensibility

### 11.5 `__init__` Methods

The `__init__` method should only assign attributes. It must not contain logic, validation, or I/O:

```python
# Good
class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, Tool] = {}

# Bad
class ToolRegistry:
    def __init__(self) -> None:
        self._tools: dict[str, Tool] = {}
        self._load_from_database()  # I/O in __init__
        self._validate_all()         # Logic in __init__
```

### 11.6 Pydantic vs Dataclass vs TypedDict

| Type | Use Case |
|---|---|
| `pydantic.BaseModel` | Request/response schemas. Configuration (`BaseSettings`). Any data that crosses an API boundary. |
| `dataclass` | Internal data containers. No validation or serialisation needed. |
| `TypedDict` | LangGraph state. Dictionary-like structures with known keys but dynamic values. |

## 12. Error Handling

### 12.1 Raise Meaningful Exceptions

Always raise exceptions with descriptive messages:

```python
# Good
raise ValueError(f"Tool '{tool_name}' is not registered.")
raise KeyError(f"Session '{session_id}' not found.")

# Bad
raise ValueError("Error")
raise Exception("Something went wrong")
```

### 12.2 Never Swallow Exceptions

```python
# Forbidden
try:
    result = tool.execute(**args)
except Exception:
    pass  # Silent swallow

# Forbidden
try:
    result = tool.execute(**args)
except:
    pass

# Good
try:
    result = tool.execute(**args)
except ValueError as e:
    logger.error("Tool execution failed: %s", e)
    raise
```

### 12.3 Use Specific Exception Types

Use built-in exceptions where appropriate. Define custom exceptions for domain-specific errors:

```python
# Good
raise ValueError("Missing 'expression' argument.")
raise KeyError("Tool not found in registry.")

# Acceptable (when no built-in fits)
class ToolExecutionError(RuntimeError):
    """Raised when a tool fails during execution."""
    ...
```

### 12.4 Validate at Boundaries

- Validate user input at the API boundary (FastAPI/Pydantic).
- Validate tool arguments at the tool boundary (each tool validates its own inputs).

Do not validate the same data in multiple places.

### 12.5 Return Early

Prefer early returns over deep nesting:

```python
# Good
def execute(self, tool_name: str, **kwargs: Any) -> Any:
    if tool_name not in self._registry:
        raise ValueError(f"Tool '{tool_name}' is not registered.")

    tool = self._registry[tool_name]
    return tool.execute(**kwargs)

# Bad
def execute(self, tool_name: str, **kwargs: Any) -> Any:
    if tool_name in self._registry:
        tool = self._registry[tool_name]
        return tool.execute(**kwargs)
    else:
        raise ValueError(f"Tool '{tool_name}' is not registered.")
```

### 12.6 Document Exceptions

Document raised exceptions in docstrings with `Raises:`:

```python
def get(self, name: str) -> Tool:
    """Look up a tool by name.

    Args:
        name: The registered tool name.

    Returns:
        The Tool instance.

    Raises:
        ValueError: If the tool is not registered.
    """
    ...
```

## 13. Logging Standards

### 13.1 Use the Project's Logging System

Never use `print()` for logging. Use the project's logging system:

```python
import logging

logger = logging.getLogger(__name__)

def execute(self, ...):
    logger.info("Executing tool: %s", tool_name)
    ...
```

### 13.2 Module-Level Logger

Every module gets a module-level logger:

```python
logger = logging.getLogger(__name__)
```

This is defined at the top of the file, after imports and before class/function definitions.

### 13.3 Log Levels

| Level | When to Use |
|---|---|
| `DEBUG` | Detailed diagnostic information for developers. |
| `INFO` | Normal operational messages (tool executed, request completed). |
| `WARN` | Unexpected but handled conditions (rate limit hit, fallback used). |
| `ERROR` | Failures that need attention but don't crash the system. |
| `CRITICAL` | System cannot continue. Immediate human intervention required. |

### 13.4 Include Trace Context

Every log line should include context for tracing:

```python
logger.info(
    "Tool executed: tool=%s session=%s duration=%.2fs",
    tool_name, session_id, duration,
)
```

### 13.5 Never Log Secrets

API keys, tokens, passwords, and personally identifiable information must never appear in logs. Redact or omit them:

```python
# Good
logger.info("API call to %s (key_id=%s)", url, key_id)

# Bad
logger.info("API call to %s with key=%s", url, api_key)
```

## 14. Documentation Standards

### 14.1 Module Docstrings

Every module must have a docstring at the top of the file explaining its purpose:

```python
"""
Jarvis AIOS
-----------
Calculator Tool

Evaluates mathematical expressions in a restricted sandbox.
"""
```

Structure: module name (one line) → blank line → description (paragraph).

### 14.2 Function Docstrings

Every public function must have a docstring. Use the Google/NumPy-style format:

```python
def execute(self, **kwargs: Any) -> Any:
    """Execute a mathematical expression.

    Args:
        expression: The mathematical expression to evaluate.
            Must be a string containing only numbers, operators,
            parentheses, and whitespace.

    Returns:
        The result of the expression as a float or int.

    Raises:
        ValueError: If the expression is missing, malformed,
            or evaluates to an error.
    """
    ...
```

### 14.3 Class Docstrings

Every class must have a docstring explaining its responsibility:

```python
class ToolRegistry:
    """
    Maintains a registry of all available tools.

    Tools are registered by name at initialisation time and
    looked up by name at execution time. The registry supports
    dynamic discovery for prompt generation.
    """
    ...
```

### 14.4 Comments Explain Why, Not What

The code itself must make *what* obvious. Comments explain *why*:

```python
# Good
# Use a restricted globals dict to prevent access to built-in functions
return eval(expression, {"__builtins__": {}}, {})

# Bad
# Evaluate the expression
return eval(expression, {"__builtins__": {}}, {})
```

### 14.5 No Stale Comments

If the code changes, the comment must change with it. Stale comments are worse than no comments.

## 15. Testing Expectations

### 15.1 Test Structure

Tests live in `app/tests/` mirroring the source structure:

```
app/tests/
├── test_Tools/
│   ├── test_calculator.py
│   ├── test_datetime_tool.py
│   ├── test_engine.py
│   └── test_registry.py
├── test_LangGraph/
│   └── test_graph.py
└── ...
```

### 15.2 Coverage Expectations

- Every public function must have at least one test.
- Every tool must be independently testable (mock the LLM, test the tool logic).
- Test the happy path, error cases, and edge cases (empty input, None values, malformed data, boundary conditions).

### 15.3 No External Dependencies

Tests must not depend on:

- Network access.
- API keys or external services.
- Live databases.
- File system state that is not created by the test itself.

If a component requires external dependencies, mock them.

### 15.4 Framework

Use `pytest` over `unittest`:

```python
# Good
def test_calculator_adds_two_numbers():
    tool = CalculatorTool()
    result = tool.execute(expression="2 + 3")
    assert result == 5

# Acceptable (when using pytest features)
@pytest.fixture
def calculator():
    return CalculatorTool()

def test_with_fixture(calculator):
    result = calculator.execute(expression="10 * 10")
    assert result == 100
```

### 15.5 Test Naming

Test functions should describe the scenario and expected outcome:

```python
def test_calculator_returns_float_for_division(): ...
def test_calculator_raises_error_for_invalid_expression(): ...
def test_calculator_raises_error_for_missing_expression(): ...
```

### 15.6 Fixtures

Use fixtures for shared setup. Fixtures should be scoped appropriately (`function` scope by default):

```python
@pytest.fixture
def registry():
    reg = ToolRegistry()
    reg.register(CalculatorTool())
    return reg

def test_registry_returns_registered_tool(registry):
    tool = registry.get("calculator")
    assert isinstance(tool, CalculatorTool)
```

### 15.7 Deterministic Tests

Tests should produce identical results on every execution.

Avoid tests that depend on:

- Timing
- External APIs
- Internet connectivity
- Randomness
- System clock (unless explicitly mocked)

## 16. Performance Guidelines

### 16.1 Measure Before Optimizing

Never optimize based on assumptions.

Profile first.

Optimize only when measurements identify a real bottleneck.

### 16.2 Readability Before Performance

Readable code is preferred over theoretical performance gains.

Small performance improvements are not worth architectural complexity.

### 16.3 Minimize LLM Calls

LLM calls are the dominant latency cost.

Reduce unnecessary calls through batching, caching, and better orchestration.

### 16.4 Cache Carefully

Cache only deterministic computations.

Document cache invalidation strategy whenever caching is introduced.

### 16.5 Prefer Lazy Evaluation

Delay expensive work until it is actually required.

### 16.6 Performance Profiling

Use profiling tools before introducing optimizations.

Document significant optimizations in pull requests.

## 17. Security Guidelines

Aligned with the Engineering Constitution (`docs/00_ENGINEERING_CONSTITUTION.md` section 4.4):

| Rule | Enforcement |
|---|---|
| No `eval()` or `exec()` on untrusted input | Code review. If unavoidable, sandbox strictly and document why. |
| No hardcoded secrets | API keys, tokens, and passwords are environment variables only. |
| Validate all user inputs | At the API boundary (FastAPI) and the tool boundary (Tool Engine). |
| Restrict filesystem access | Tools must not read arbitrary paths. Approved directories only. |
| Never trust LLM output blindly | Validate structure, types, and bounds before using LLM responses. |
| Never log secrets or PII | Secrets must be redacted before logging. |
| Never expose API keys | API keys in error messages, logs, or responses are a security incident. |

Any change that touches authentication, authorisation, secrets management, file access, code execution, or network access must be reviewed by a human with security expertise.

## 18. Allowed Patterns and Forbidden Patterns

### 18.1 Allowed Patterns

| Pattern | Description | Example |
|---|---|---|
| Dependency Injection | Dependencies are passed in through the constructor. | `ToolEngine(registry: ToolRegistry)` |
| Global Singleton (documented) | Module-level instance with a clear, single purpose. | `jarvis = Jarvis()`, `registry = ToolRegistry()` |
| Composition | Objects contain other objects they delegate to. | `ToolEngine` uses `ToolRegistry` |
| Interface Inheritance | Inheriting from abstract base classes. | `class CalculatorTool(Tool):` |
| Factory Functions | Functions that construct complex objects. | `def create_engine() -> ToolEngine:` |
| Early Returns | Return early instead of deep nesting. | See section 12.5. |
| Type Aliases | Named aliases for complex types. | `ToolMap: TypeAlias = dict[str, Tool]` |
| Context Managers | `with` statements for resource lifecycle. | `with redirect_stdout(output):` |
| Dataclasses | `@dataclass` for internal data containers. | See section 9.4. |
| Interface-Based Design | Depend on abstractions instead of implementations. | `LLMClient` interface |
| Feature Flags | Experimental features remain isolated. | `ENABLE_STREAMING` |

--- 

### 18.2 Forbidden Patterns

| Pattern | Why | Example |
|----------|-----|---------|
| `eval()` on untrusted input | Remote code execution | `eval(user_input)` |
| `exec()` on untrusted input | Remote code execution | `exec(user_code)` |
| Wildcard imports | Hidden dependencies | `from module import *` |
| Circular imports | Indicates architectural problems | Import loops |
| Hardcoded paths | Break portability | `/Users/...` |
| Hardcoded secrets | Security risk | API keys in source |
| Global mutable state | Breaks session isolation | Shared dictionaries |
| Silent exception handling | Hides failures | `except: pass` |
| Business logic in FastAPI routes | Violates architecture | Complex processing in endpoints |
| Business logic in Runtime | Runtime should orchestrate only | Processing business rules |
| Business logic in LangGraph nodes | Nodes orchestrate only | Tool logic inside nodes |
| Direct tool execution | Bypasses Tool Engine | `tool.execute()` |
| LLM calls inside tools | Breaks architectural boundaries | Calling provider directly |
| Deep inheritance | Difficult to maintain | A → B → C → D |
| Magic numbers | Difficult to understand | `if retries > 3` |
| Unjustified `Any` | Weakens static typing | `Any` without explanation |

## 19. Code Review Checklist

Every pull request must satisfy the following before it can be merged:

### Code Quality

- [ ] Code follows all standards in this document.
- [ ] Type hints are complete and correct.
- [ ] Docstrings are present for all public APIs.
- [ ] No dead code, commented-out code, or placeholder stubs.
- [ ] Imports are clean, explicit, and correctly grouped.
- [ ] No magic numbers or strings — constants are used.

### Architecture

- [ ] Architecture is preserved (no layer violations).
- [ ] No dependencies violate the allowed/forbidden table (docs/02_ARCHITECTURE.md section 7).
- [ ] No bypassing of Runtime, LangGraph, Tool Engine, or Registry.
- [ ] Session isolation is maintained.
- [ ] Public APIs remain backward compatible unless intentionally changed.
- [ ] No unnecessary third-party dependencies introduced.
- [ ] Documentation updated when behavior or architecture changes.

### Security

- [ ] No secrets are exposed.
- [ ] No unsafe `eval()`/`exec()` on untrusted input.
- [ ] No hardcoded credentials, API keys, or tokens.
- [ ] All user inputs are validated.
- [ ] LLM output is validated before use.

### Testing

- [ ] Tests exist for all new public functions.
- [ ] Tests pass.
- [ ] Edge cases and error paths are covered.
- [ ] Tests do not depend on external services or API keys.

### Logging

- [ ] Logging is appropriate (no `print()`).
- [ ] Trace context is included (`session_id`, `request_id`).
- [ ] No secrets or PII are logged.

### Error Handling

- [ ] No silent exception swallowing.
- [ ] Exceptions are raised with meaningful messages.
- [ ] Input validation exists at all boundaries.

- Predictable

- Secure

### Versioning

- **Major** — Changes to Core Standards or Forbidden Patterns.

### Transition Period

When these standards are updated, existing code that violates the new rules is flagged as technical debt. It must be resolved within two sprints or be explicitly exempted with a documented rationale.
